    -> Infrastructure As A Service - IAAS 

        -> AWS Global Infrastructure 

            -> Amazon Cloud Computing resources (servers, storage, databases etc) 
            are available across the world. 
            In simple terms if we see this https://aws.amazon.com/about-aws/global-infrastructure/ 
            then Amazon Data Centers are available in different geographical locations.

        -> In terms of AWS, infrastructure is made up of 

            -> Regions 

            -> Availability Zones 

            -> Edge locations

        -> Regions

            -> Regions are designed to service AWS customers (or your users) 
                that are located closest to a region.

            -> Availability of Services are different for different AWS Regions.

            -> Price varies from region to region

    -> Availability zone (AZ)

        -> Availability zone is where the actual data centers are located

        -> Within a Region, there can be multiple Availability zones

        -> Each AZ has independent Power Supply, Networking, Cooling System, Physical Security.

        -> https://www.researchgate.net/profile/Ivan_Beres/publication/360541079/figure/fig1/AS:1154856056291328@1652350548135/AWS-Regions-and-Availability-Zones-Scott-nd.ppm


Class - 4
==========

    -> Edge locations

        -> An AWS Edge Location is a data center in the AWS global network used by services 
        like CloudFront, Global Accelerator, and Lambda@Edge to store cached content 
        (images, videos) and process requests closer to end-users, 
        drastically reducing latency by serving data from the nearest point of presence(pop) 
        rather than a distant main AWS Region. 
        They act as caching warehouses, improving performance for websites.

    -> VPC (Virtual Private Cloud)

        -> A Virtual Private Cloud is a secure, isolated private cloud hosted 
        within a public cloud. 

        -> VPC customers can run code, store data, host websites & applications in isolated mode

        -> NOTE: Having VPC is similar to having your own data center inside AWS. 

        -> VPC is a service that lets customers(IBM) provision a logically isolated 
        section of the AWS Cloud where you can launch AWS resources(Servers) 
        in a VIRTUAL NETWORK that you define.

    -> VPC Architecture Reference 

        -> https://media.licdn.com/dms/image/v2/D4D12AQFI8sqvP4xFFQ/article-cover_image-shrink_720_1280/article-cover_image-shrink_720_1280/0/1700537758681?e=2147483647&v=beta&t=zE97TFGw5FQJEjUksqx7SohNsqqV57-IYQ_pj7K2M2U

    -> VPC components

        -> Internet Gateway (Gate For Building)

            -> An Internet Gateway must be attached to a VPC if the resources inside the VPC 
                need to connect to resources via the open internet.

        -> Subnets (Floors In Building)

            -> A subnet or sub network is an logical subdivision of an Amazon Virtual Private Cloud (VPC)

        -> Route Tables (Lift In Building)

            -> A route table contains a set of rules called routes, 
                that are used to determine where network traffic is directed.

        -> Network Access Control List - NACL (Watchman For Floor)

            -> subnet-level firewall that acts as an optional security layer 

        -> Security Group (Watchman For Flat)

            -> server-level firewall which is mandatory

	-> Edify is going to setup the LMS App environment i.e servers on AWS, 
        their clients are from XYZ (REGION) Location

	-> First Subnet is for web servers, around 4k

	-> Second Subnet is for backend servers, around 4k

	-> Third Subnet is for database servers, around 4k

	-> In the future, they might add more number of subnets i.e cache subnets, load balancer subnet etc

	-> The web servers and app servers needs to be communicated from internet

	-> The database servers needs to be communicated only from App Servers, but not from internet
    
	-> Coming to security of Web Servers, they should have only 
		-> SSH traffic enabled for administration from internet
		-> HTTP traffic for clients from internet

	-> Coming to security of App Servers, they should have only 
		-> SSH traffic enabled for administration from internet
		-> HTTP traffic port 8080 for clients from internet

	-> Coming to security of Database Servers, they should have only 
		-> SSH traffic enabled from web servers only
		-> Postgres Database Traffic Allowed from App Servers

Class - 10
==========

    -> Storage 

        -> AWS provides three popular services for Storage
                
            -> Elastic Block Store (EBS)
            -> Elastic File System (EFS)
            -> Simple Storage Service (S3)

    
    -> Elastic Block Store (EBS)

        -> Elastic Block Store (EBS) - Used for Server Storage (OS(ubuntu), Softwares(apache))
            
            -> It cannot be shared across multiple zones i.e multiple data centers

        -> EBS Must have an Root Volume (OS Disk i.e Primary Disk)

        -> EBS Optionally can have an Additional Volumes (Secondary Disk)

        -> Based On the Zone, Volume will be created in the same zone where instance is launched


        -> EBS - Limitations 

            -> Using EBS we cannot get the data directly across Multiple Availability Zones
            -> Using EBS we cannot connect to MULTIPLE INSTANCES at same time 
            -> With EBS Volumes, you can only scale up, but NO SCALING DOWN
            -> EBS Volumes can scale only up to TB's

Class - 12
==========

    -> S3 - Simple Storage Service 

        -> https://aws.amazon.com/s3/

        -> Object storage built to retrieve any amount of data from anywhere

        -> S3 Buckets

            -> Data is stored in Buckets, Buckets are the main storage containers of S3.

        -> S3 Objects

            -> Objects are the fundamental entities stored in Amazon S3. An Object is a file.

        -> You can store unlimited objects in a bucket but an object cannot exceed 5 TB

        -> By default, all buckets & objects are private.

        -> S3 provides high availability, Objects are redundantly stored on multiple devices 
            across multiple facilities ( Multiple AZ's)

        -> Storage Classes

            -> A Storage Class represents the classification assigned to each object in S3.

            -> Each storage class has different Storage cost
            
            -> https://aws.amazon.com/s3/storage-classes/

        -> S3 Lifecycle Policy 

            -> An object lifecycle policy is a set of rules that automate the 
            transitioning of an object's storage class to a different storage class 
            (or deletion) based on specified time intervals.

        -> S3 Versioning 

            -> S3 versioning is a feature to manage and store versions of an object

            -> S3 versioning protects your data against accidental overrides and deletion.

        -> S3 Static Web hosting
            
            -> You can use Amazon S3 to host a static website.

        "securely control access" to AWS resources.

            -> IAM Policies

        -> IAM Users 

            -> Root User: The user created when you first create your AWS account 
            is called the "root" user -> your_name@mail.com -> registered with aws account 
            root user has FULL rights i.e no Restrictions 

            -> IAM User: represents an AWS user that you may create (in IAM), 
            who will have "varying degrees of access"(read/write) to the AWS account.
            By default IAM users will not have any permissions.
            IAM Users can access aws resources using 

                    -> AWS Management Console: Human Based with Browser Login - username & password 

                    -> AWS Programmatic Access: Softwares & Tools Based Login - access & secret key  

            -> IAM Policy

                -> A policy is an object in AWS that defines permissions. 

            -> IAM Group: 
            
                -> It's an logical collection of IAM Users, used for organizing 
                permissions in easy way.

Class - 18
==========

    -> Security For Instances

        -> So when we are talking about "increased security", what we want to focus on is, 
        placing of EC2 instances that are holding our data in "private subnets".

    -> However this causes some issues

        -> We cannot serve traffic from private instances as there is no route 
        to the open internet and also we cannot access or ssh into 
        EC2 instances that are in private subnets

            -> Solution for above is - Bastion Server / Jump Server 

    -> Bastion Host / Server 

        -> A Bastion Host is an EC2 instance that lives in a public subnet, 
        and is used as a "gateway" for traffic that is destined for instances 
        that live in private subnets.

        -> A bastion host acts as a "portal" to access EC2 instances 
        that are located in a private subnet.
